# Агент сбора экспертных знаний — прототип

Closed-loop сбор знаний: агент ведёт текстовое интервью с экспертом →
транскрипты превращаются в структурированную базу → база влияет на следующие
интервью (агент спрашивает про пробелы и противоречия, а не повторяется).

Прототип реализует **контур сбора** целиком. Контур знаний (RAG/LoRA) не
реализован, но `extracted_items` и `topic_summaries` уже в том формате, из
которого он собирается позже без переделок.

## Архитектура

Тонкий врапер + PostgreSQL + OpenAI-совместимый LLM-эндпоинт. Один агент = одна
модель + тулы + луп; побочные эффекты — через tool calls. Постобработка — три
оффлайн-джобы. Ни `temperature`, ни `max_tokens` не отправляются — дефолты
сервера решают сами.

```
Telegram (aiogram) ─▶ agent.run_turn ─▶ OpenAI-compatible /v1 (chat + tools) ─▶ Postgres
                                                                                  │
                          оффлайн: extract → summary → eval ◀────────────────────┘
                          эмбеддинги: локальный harrier /v1/embeddings ──▶ pgvector
```

| Файл | Что делает |
|------|-----------|
| `src/prompts.py` | Ядро продукта: методология интервью, промпты экстракции/сводки/евала |
| `src/agent.py` | Один ход интервью: контекст → луп тулов → запись; токен-бюджет |
| `src/state.py` | Компактный STATE-блок (план + выжимка сводки + auto-RAG из базы + бюджет) |
| `src/tools.py` | `update_plan` / `mark_covered` / `search_knowledge` / `web_search` / `web_fetch` / `end_session` |
| `src/embed.py` | Локальные harrier-эмбеддинги (опционально) для дедупа и `search_knowledge` |
| `src/websearch.py` | Tavily: проверка фактов о железе/терминах прямо в рассуждении (опционально) |
| `src/llm.py` | OpenAI-совместимый клиент; без `temperature`/`max_tokens`; луп тулов |
| `src/jobs/extract.py` | Транскрипт → факты/Q&A/термины с цитатами; CAS + pgvector-дедуп |
| `src/jobs/summary.py` | Пересборка сводки по теме **с нуля** из канонических записей |
| `src/jobs/eval.py` | Batch-судья по вопросам агента |
| `scripts/selftest.py` | Симулированное интервью: LLM играет эксперта, гоняет реальный агент end-to-end |
| `scripts/stats.py` | Per-session наблюдаемость: токены, тулы, вердикты судьи |
| `scripts/view.py` | Просмотр извлечённого (ловля тихой порчи данных) |
| `scripts/export.py` | Экспорт базы знаний темы в Markdown (бизнес-артефакт) |

## Модели

**Чат-LLM** — любой OpenAI-совместимый `/v1` сервер (`OPENAI_BASE_URL`): vLLM,
llama.cpp, LM Studio, TGI. `DIALOG_MODEL` дёргается на каждый ответ эксперта;
`EXTRACT/SUMMARY/EVAL_MODEL` — оффлайн, могут указывать на другую (дешёвую)
модель того же эндпоинта. `temperature` и `max_tokens` не шлются.

**Эмбеддинги** — локальный `microsoft/harrier-oss-v1-270m` (Gemma3-270m энкодер,
dim 640, last-token pooling, L2-norm, cosine). Веса уже скачаны в
`./models/harrier-oss-v1-270m` (или `python -m scripts.download_model`). Подаётся
через text-embeddings-inference как `/v1/embeddings`, URL в `EMBED_BASE_URL`.

> harrier — **эмбеддинг-модель, интервью вести не может**. Диалог требует
> отдельной чат/instruct-модели на `OPENAI_BASE_URL`.

## Запуск

```bash
cp .env.example .env          # TELEGRAM_BOT_TOKEN + OPENAI_BASE_URL (чат-сервер)
docker compose up --build     # поднимает Postgres (миграция автоприменяется) + бота
```

Из контейнера `localhost` — это сам контейнер; чтобы дотянуться до сервера на
хосте, ставь `OPENAI_BASE_URL=http://host.docker.internal:8000/v1`.

Эмбеддинги (опционально) — поднять harrier через text-embeddings-inference:

```bash
# пример; порт/девайс под своё железо
text-embeddings-router --model-id ./models/harrier-oss-v1-270m --port 8080
# затем в .env: EMBED_BASE_URL=http://host.docker.internal:8080/v1
```

Локально без Docker:

```bash
pip install -r requirements.txt
# поднять Postgres (с pgvector), применить migrations/001_init.sql, выставить DATABASE_URL
python -m src.bot
```

Прод-запуск на Windows без Docker — watchdog, переживает падения:

```powershell
powershell -File scripts\run_bot.ps1
```

Оффлайн-джобы вручную (обычно бот запускает extract+summary+eval сам по завершении):

```bash
python -m src.jobs.extract <session_id>
python -m src.jobs.summary <topic>
python -m src.jobs.eval    <session_id>
python -m scripts.view     <topic>
python -m scripts.stats               # токены/тулы/качество по сессиям
python -m scripts.export   <topic>    # база знаний -> kb_<topic>.md
```

Самотест без живого эксперта (LLM играет интервьюируемого, сессии идут в
отдельную тему `selftest` и не пачкают боевую базу):

```bash
python -m scripts.selftest --turns 8 --postprocess
```

## Веб-поиск (опционально)

`TAVILY_API_KEY` в `.env` включает тулы `web_search`/`web_fetch`: агент проверяет
факты о железе/терминах прямо в ходе рассуждения (существует ли модель, какой в
ней чип), прежде чем строить на них вопрос — вместо галлюцинированной
предпосылки. Tavily отдаёт синтезированный ответ + выжатый контент страниц, а не
десять синих ссылок. Без ключа тулы скрыты от модели, ничего не ломается.
Успешные запросы кэшируются в памяти процесса (бесплатный тариф ~1000 req/мес).

## Демо closed loop

1. `/start ml-infra` — интервью по теме `ml-infra`, эксперт отвечает ~15 мин,
   `/finish` (или агент сам зовёт `end_session`).
2. `python -m scripts.view ml-infra` — таблица фактов/Q&A с цитатами + сводка.
3. Второй эксперт: `/start ml-infra` — интервью начинается уже с учётом сводки
   (открытые вопросы, «ваш коллега утверждал X — согласны?»). Это видимое
   доказательство замкнутой петли.

## Эмбеддинги (опционально)

Дедуп при вставке и `search_knowledge` работают на локальном harrier через
OpenAI-совместимый `/v1/embeddings`. Без `EMBED_BASE_URL` контур сбора всё равно
бежит: каждый факт канонический, `search_knowledge` пусто, дедуп отключён. С
эндпоинтом — pgvector считает косинусную близость: близко → `duplicate_of` +
инкремент `confirmation_count`; на границе → LLM-проверка на противоречие;
далеко → новая каноническая запись. Запросы `search_knowledge` получают
instruct-префикс harrier, хранимые факты эмбедятся без него.

## Принципы схемы

`messages` — сырьё навсегда, не мутируется. Всё остальное — производные слои,
перегенерируемые новым парсером. Каждый `extracted_item` обязан иметь цитату и
`source_message_id` — любой факт проверяем за секунду. Поле `origin`
(`expert_claim` / `confirmed_hypothesis`) структурно отделяет утверждения
эксперта от подтверждённых гипотез агента; неявные реакции не извлекаются как
факты вовсе — защита от тихой порчи сводки и петли.

## Известные риски

| Риск | Смягчение (в коде / план) |
|------|---------------------------|
| Ложные факты из вежливого неоспаривания | Поле `origin`: факт только из `expert_claim`; гипотезы — слабый вес; неявное → открытый вопрос |
| Дубли фактов между сессиями | pgvector-дедуп при вставке: `duplicate_of` + `confirmation_count`; сводка читает только канонические |
| Разрастание state_block | Бюджет ≤800 токенов, выжимка сводки, остальное через `search_knowledge` |
| Latency диалоговой модели | typing-индикатор; лёгкая модель на `DIALOG_MODEL`; мерить на живых прогонах |
| Самокритика слабее внешнего критика | `jobs/eval.py` обязателен, не опция; судья получает пару «ответ → вопрос» |
| Тихая порча в экстракции | цитата + `source_message_id` + `origin` + `scripts/view.py` |
| Модель забывает дёргать тулы | coverage advisory; врапер не ломается; `search_knowledge` — подстраховка |
| Двойной запуск экстракции | Статус-машина + атомарный CAS `finished→extracting` |
| Неконтролируемая стоимость сессии | Токен-бюджет: софт-кап «закругляйся» → хард-кап принудительный `end_session` |
| Смешение версий промптов | `prompt_version` на всех LLM-производных записях |

## Что осознанно НЕ сделано (скоуп прототипа)

- STT/голос — отдельный этап.
- RAG/LoRA-контур — данные готовы (`extracted_items` + `topic_summaries`,
  Q&A-пары как чанки/SFT), контур не реализован.
- Тест-сет из 10–15 пар «ответ → вопрос» для евала (Этап 2) — размечать вручную.
- **Граф знаний (GraphRAG) — осознанно нет.** На текущем масштабе (десятки–сотни
  канонических записей на тему) типизированные записи + pgvector + связи
  `contradicts`/`duplicate_of`/`confirmation_count` уже образуют неявный граф и
  закрывают все реальные запросы. Явный граф окупается на тысячах сущностей с
  multi-hop-вопросами («какие эксперты расходятся по параметру X у материала Y
  на принтере Z»); строить его раньше — платить за экстракцию сущностей и
  поддержку схемы без выигрыша в качестве. Пересмотреть, когда база перерастёт
  ~несколько тысяч записей или появятся multi-hop-запросы.
